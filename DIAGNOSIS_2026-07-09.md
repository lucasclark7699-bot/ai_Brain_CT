## 二、性能瓶颈分析（你感知最明显的"慢"）

### 2.1 测试连接慢 —— 根因：没有超时控制

`src/api_client.py` 第 35–38 行：

```python
self.client = OpenAI(
    base_url=base_url,
    api_key=api_key,
    # ❌ 没设 timeout、没设 max_retries
)
```

- OpenAI SDK 默认 `timeout=600秒`、`max_retries=2`。
- 一旦供应商响应慢或网络抖动，一次 `test_connection`（第 182 行）可能卡很久或反复重试，**用户感知就是"点了没反应/很久才返回"**。
- `test_connection` 本身只发 1 个 `max_tokens=5` 的请求，逻辑没问题，**慢来自无超时兜底**。

**修复方向（待你拍板）**：`OpenAI(timeout=15, max_retries=1)`，或给测试连接单独设更短的超时。

### 2.2 模型验证慢 —— 根因：串行 5~7 次请求，其中 2 次完全无效 ⭐重点

`src/panels/model_verification.py` 的 `verify_model_endpoint()` 一次验证串行发起：

| 步骤 | 代码位置 | 请求数 | 说明 |
|------|----------|--------|------|
| 模型名一致性 | 第 216 行 | 1 | 必要 |
| 数数测试 | 第 245 行 | 1 | 必要 |
| 数数重试（失败才发） | 第 262、280 行 | 最多 2 | 容错，可保留 |
| **上下文"预热"** | **第 314–315 行** | **2** | **❌ 完全无效，返回值直接丢弃** |
| 上下文实际测试 | 第 322 行 | 1 | 必要 |

- **正常情况 = 5 次请求，最坏 = 7 次**，且全部**串行**（一个接一个等返回）。
- **第 314–315 行是纯浪费**：
  ```python
  for message in context_messages:   # "记住数字7" / "记住数字3"
      _chat_with_raw_response(client, message ..., max_tokens=5)
      # ↑ 返回值没接任何变量，发出去就丢了
  ```
  普通/中转 API 是**无状态**的，单独发"记住数字7"并不会让后续请求"记住"——只有第 322 行把完整上下文一起发的 `context_prompt` 才有效。这两发等于白烧 token + 白等两轮网络往返。
- 叠加 2.1 的"无超时"，单次慢则整体慢，用户感知"验证返回特别慢"。

**修复方向（待你拍板）**：
1. **直接删掉第 314–315 行那段预热循环**（光发 `context_prompt` 即可，省 2 次请求）。
2. 给客户端加 `timeout`。
3. 若想更快，可考虑用线程并发发独立请求（但中转 API 并发可能被限流，需谨慎）。

### 2.3 其他性能隐患（你没提，但相关，列出来备着）

| 位置 | 问题 | 影响 |
|------|------|------|
| `src/panels/star_map.py` | 纯 Python 力导向布局（O(n²) × 100 轮迭代），无 `@st.cache_data` | 对话多/关键词多时，切到星空图会卡 |
| `src/panels/model_diagnostics.py` | 每次切 tab 重算相似度，无缓存 | 重复计算 |
| `src/panels/decay_curve.py` | 每次切 tab 重算 TF-IDF 相似度，无缓存 | 重复计算 |
| `src/analyzer.py` | 相似度用 TF-IDF + 余弦，对话多时 O(n²) | 尚可，但无缓存会放大 |

**小结**：性能瓶颈集中在「网络请求无超时 + 模型验证有无效请求」，图表类主要是缺缓存。前者解决后"慢"的体感会大幅改善。

## 三、功能板块逻辑问题（不符合常理，需精调）

### 3.1 ⭐「记忆衰减曲线」概念硬伤（最严重）

`src/panels/decay_curve.py` 把"记忆准确度"定义为 **"当前对话 vs 历史的 TF-IDF 相似度"**。

- **问题**：用户一**换话题**，相似度立刻骤降 → 曲线"断崖式下跌" → 被系统标记为"记忆衰减 / 幻觉风险上升"。
- 但**换话题 ≠ 记忆衰减**。这俩是完全不同的事。逻辑上把"话题一致性"误当成"记忆准确度"，**不符合常理**，容易让使用者被误导。
- **精调方向（待你拍板）**：要么重新定义指标（如用"模型是否能在后续召回早期事实"来测真实记忆），要么在 UI 上明确标注"这是话题一致性趋势，非记忆准确度"，改名避免误导。

### 3.2 「上下文记忆测试」逻辑无效

就是 2.2 里说的第 314–315 行：想测"记忆"，却用无状态 API 单独发两次"记住数字"去"预热"。**无状态接口根本记不住**，这段测试在逻辑上测不出记忆能力，只测了"这一次把上下文一起发能不能算对 7+3=10"。

- 建议：去掉预热，直接评测 `context_prompt`（已包含完整上下文的那条）的回答即可；若真要测"记忆"，需依赖对话存储（本项目的 `conversations` 表）跨轮召回，而非指望 API 自带的上下文窗口。

### 3.3 预警中心统计重复查询

`src/panels/alerts_panel.py` 第 66–70 行已查过 `alerts`（列表，limit=50），第 78 行又 `for a in get_alerts(limit=1000)` **再查一遍全量**算严重程度分布。

- 问题：① 重复 IO；② 统计口径（全量 1000）与显示列表（50）不一致，饼图和列表数字对不上。
- 建议：一次查询复用，或用 SQL 聚合直接出统计。

### 3.4 自动预警钩子无容错

`alerts_panel` 第 52 行声称"新消息后自动扫描"，`chat_panel` 里新消息后确实调了 `detect_contradictions`/`extract_keywords`，但**这些调用没有 try/except**（之前排查已发现）。一旦分析函数异常，会拖垮整条对话流程（虽然上一轮已加面板级隔离，但最好在分析调用处就近兜底）。

### 3.5 各面板"记忆"概念不统一

星空图（关键词关联）、热力图（logprobs 注意力）、衰减曲线（TF-IDF 相似度）、诊断（token/延迟）、验证（真伪）…… 都叫"记忆/注意力"，但底层数据源和定义各说各话。使用者容易困惑"到底在测什么"。

- 建议：在 UI 文案上统一术语，或加一行"本图测量的是 XXX"，降低理解成本。
